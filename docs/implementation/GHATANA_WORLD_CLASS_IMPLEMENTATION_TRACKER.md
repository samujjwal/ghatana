# 3. Critical Findings

## P0 — Committed release evidence is for the wrong commit

The Data-Cloud release bundle inside `efdcdd4...` says `pass: true`, but its `commit`, `sourceCommitSha`, and `targetCommitSha` are all `936924cf915e366f3f0c47bee266a277601b1288`.  The product release readiness payload repeats the same target/source commit mismatch while claiming validated readiness. 

**Impact:** Every downstream “pass” claim in the committed release bundle is stale for this audit target. The release evidence cannot prove Data-Cloud, Action Plane, agent, AI governance, operations, or route lifecycle correctness for `efdcdd4...`.

**Classification:** `BROKEN` / `STALE_OR_INVALID`.

---

## P1 — Batch delete dry-run token cannot validate

`EntityCrudHandler.buildBatchDeleteToken()` signs a payload containing tenant, collection, **count**, and issued-at timestamp. `validateBatchDeleteToken()` recomputes the expected HMAC from tenant, collection, and issued-at timestamp but omits `count`. Therefore, a token generated by the dry-run path will not match the execute-path validator. 

**Impact:** The destructive batch-delete journey is broken:

```text
dry-run → confirmation token → execute delete
```

The execute step rejects the token with signature mismatch.

**Classification:** `BROKEN`.

---

## P1 — Pattern lifecycle is incomplete versus required lifecycle

The prompt requires lifecycle coverage for draft, candidate, shadow, recommended, approved, active/predictive, degraded, retired, and rollback. The production validator accepts only `active`, `draft`, `disabled`, `shadow`, `recommended`, and `predictive`; it omits `candidate`, `approved`, `degraded`, `retired`, and `rollback`. 

`PatternSpecCompiler.compileTyped()` validates through `PatternSpecValidator.validate(spec.toMap())` without passing production environment, commit SHA, or registry context, so production-specific lifecycle/evidence constraints are not guaranteed at the compiler boundary. 

**Impact:** Pattern lifecycle governance cannot be marked complete end to end.

**Classification:** `PARTIAL`.

---

## P1 — Integration and API contract tests are not release-blocking

The active Data-Cloud module evidence classifies `delivery:api-contract-tests` and `integration-tests` as advisory with no release check task, while core production modules are release-blocking. 

**Impact:** Core E2E journeys can pass release without the integration-test modules being enforced as release-blocking. This violates the audit requirement that E2E journeys be proven by meaningful tests/release gates.

**Classification:** `WIRED_NOT_TESTED` / `BUILD_OR_RELEASE_GATE_GAP`.

---

## P1 — Audio-Video and YAPPC are in scope but not proven by the Data-Cloud release evidence

The generated settings include Audio-Video and YAPPC modules in the same repo snapshot.  However, the product release readiness artifact is scoped to affected product `data-cloud`, not Audio-Video/YAPPC. 

**Impact:** The requested audit scope includes Audio-Video and YAPPC/Data-Cloud integration, but committed release evidence does not prove those journeys at the requested commit.

**Classification:** `UNKNOWN_BLOCKED_BY_MISSING_EVIDENCE`.
