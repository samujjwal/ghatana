# Getting Started with YAPPC

**Target audience:** Product managers, developers, and architects starting their first YAPPC project  
**Estimated reading time:** 10–15 minutes  
**Last updated:** 2026-04-01

---

## What is YAPPC?

YAPPC (Yet Another Product Platform Creator) is an **AI-native product scaffolding platform** that guides a cross-functional team through eight lifecycle phases — from raw idea capture right through to deployed, observed, and evolving software.

Each phase is backed by:
- **AI suggestions** — contextual guidance generated by `AISuggestionService`
- **Phase gates** — validated by `PhaseGateValidator` before every transition
- **Human approvals** — durable approval workflow for critical milestones
- **Observability** — every action is traced, metered, and logged

---

## The Eight Lifecycle Phases

```
INTENT → SHAPE → VALIDATE → GENERATE → RUN → OBSERVE → LEARN → EVOLVE
```

| Phase | Purpose | Key artefacts produced |
|-------|---------|----------------------|
| **INTENT** | Capture the product idea in plain language | Intent record, enriched requirements |
| **SHAPE** | Design the system — domain model, APIs, architecture | Domain model, API spec, architecture outline |
| **VALIDATE** | Validate requirements and design against policies | Validation report, policy decisions |
| **GENERATE** | Scaffold code, config, CI pipelines | Source tree, Dockerfile, Kubernetes manifests |
| **RUN** | Deploy and observe first run | Deployed artifact, run report |
| **OBSERVE** | Monitor production behaviour | Metrics, traces, anomaly alerts |
| **LEARN** | Distil learnings into the knowledge graph | Learning items, ADRs |
| **EVOLVE** | Plan and execute the next iteration | Backlog, roadmap update |

## Tutorial 1 — Your First Project (5 minutes)

This tutorial walks you through creating a new project and reaching the SHAPE phase.

### Prerequisites

- Access to a YAPPC instance (local: `http://localhost:8080`)
- An API key (ask your admin) or a valid user account

### Step 1 — Authenticate

```bash
# Request a token
curl -s -X POST http://localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"you@example.com","password":"yourpassword"}' \
  | jq .token
```

Store the returned JWT:

```bash
export YAPPC_TOKEN="<token from above>"
```

### Step 2 — Capture Intent

Describe your product idea in plain language. YAPPC accepts free-form text and uses AI to enrich it.

```bash
curl -s -X POST http://localhost:8080/api/v1/yappc/intent/capture \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "projectId": "my-first-project",
    "content": "A web application that helps small teams track daily stand-up notes and decisions, with AI summarisation and Slack integration.",
    "persona": "PRODUCT_MANAGER"
  }' | jq .
```

A **Suggestion Panel** will also appear in the frontend UI with AI recommendations of type `REQUIREMENT`, `DESIGN`, `TEST`, `RISK`, or `ACTION` to help refine the intent.

### Step 3 — Analyse Intent with AI

Send the captured intent through the LLM pipeline for decomposition:

```bash
curl -s -X POST http://localhost:8080/api/v1/yappc/intent/analyze \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"intentId": "<id from previous response>"}' \
  | jq .
```

### Step 4 — Advance to SHAPE Phase

Before advancing, YAPPC runs three gate checks:
1. **Entry criteria** for SHAPE are satisfied
2. **Exit criteria** for INTENT are satisfied  
3. **Required artefacts** for INTENT are present

```bash
curl -s -X POST http://localhost:8080/api/v1/lifecycle/advance \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "projectId": "my-first-project",
    "targetPhase": "SHAPE",
    "conditions": {
      "intent.captured": true,
      "intent.reviewed": true
    }
  }' | jq .
```

If the gate passes you receive `{ "success": true, "targetPhase": "SHAPE" }`.  
If a blocker exists the response is HTTP 409 with a `blockers` array explaining what is missing.

### Step 5 — Derive System Shape

```bash
curl -s -X POST http://localhost:8080/api/v1/yappc/shape/derive \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"intentId": "<intent-id>"}' \
  | jq .
```

This triggers the domain-model and architecture-outline artefacts to be generated. Once they appear in the artefact store the gate for advancing to VALIDATE will open.

## Tutorial 2 — Working with AI Suggestions

The `SuggestionPanel` (accessible in the YAPPC frontend `ui/core/SuggestionPanel`) fetches phase-contextual suggestions from the AI backend.

### Triggering Suggestions via the API

```bash
curl -s -X POST http://localhost:8080/api/v1/projects/my-first-project/suggestions \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "phase": "SHAPE",
    "context": {
      "title": "Stand-up notes tracker",
      "stakeholders": "product team, engineering"
    }
  }' | jq .
```

### Suggestion Types

| Type | Meaning | Example |
|------|---------|---------|
| `REQUIREMENT` | Missing or unclear requirements | "Clarify offline-first behaviour requirement" |
| `DESIGN` | Architectural or design concern | "Extract Slack integration as a separate adapter module" |
| `TEST` | Test coverage gap | "Add negative test cases for empty stand-up input" |
| `RISK` | Dependency or delivery risk | "Slack API rate limits may affect real-time sync" |
| `ACTION` | Concrete next step | "Schedule architecture review before VALIDATE phase" |

### Dismissing a Suggestion

```bash
curl -s -X DELETE \
  http://localhost:8080/api/v1/projects/my-first-project/suggestions/<suggestion-id> \
  -H "Authorization: Bearer $YAPPC_TOKEN"
```

## Tutorial 3 — Human Approval Workflow

Certain phase transitions require explicit approval from a designated reviewer (e.g. advancing from VALIDATE to GENERATE requires a sign-off from the architecture lead).

### Submit an Approval Request

The lifecycle service submits approval requests automatically when a gate requires human signoff. The request appears in the approver's queue:

```bash
curl -s http://localhost:8080/api/v1/approvals/pending \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  -H "X-Tenant-Id: my-tenant" \
  | jq .
```

### Approve a Request

```bash
curl -s -X POST http://localhost:8080/api/v1/approvals/<approval-id>/approve \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  -H "X-Tenant-Id: my-tenant" \
  -H "Content-Type: application/json" \
  -d '{"notes": "Architecture looks solid. Approved to proceed."}' \
  | jq .
```

### Reject a Request

```bash
curl -s -X POST http://localhost:8080/api/v1/approvals/<approval-id>/reject \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  -H "X-Tenant-Id: my-tenant" \
  -H "Content-Type: application/json" \
  -d '{"notes": "Missing API contract for the Slack webhook endpoint."}' \
  | jq .
```

## Tutorial 4 — Running the Phase Gate Validator Directly

You can call the gate validator independently without attempting a phase advance. This is useful for pre-flight checks in CI pipelines.

```bash
# Example: check whether VALIDATE → GENERATE gate is open
curl -s -X POST http://localhost:8080/api/v1/lifecycle/advance \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "projectId": "my-first-project",
    "targetPhase": "GENERATE",
    "conditions": {
      "validation.completed": true,
      "policy.approved": true
    }
  }' | jq '{success, blockers, artifactBlockers, criteriaBlockers}'
```

A `success: false` response with populated `blockers` tells you exactly what is missing — no guesswork needed.

## Tutorial 5 — Workflow Templates

YAPPC ships several durable workflow templates for common coordination patterns.

### List Available Templates

```bash
curl -s http://localhost:8080/api/v1/workflows/templates \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  | jq .
```

### Start a Workflow

```bash
curl -s -X POST http://localhost:8080/api/v1/workflows/lifecycle-advance/start \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  -H "X-Tenant-Id: my-tenant" \
  -H "Content-Type: application/json" \
  -d '{
    "projectId": "my-first-project",
    "targetPhase": "GENERATE"
  }' | jq .
```

### Poll for Completion

```bash
curl -s http://localhost:8080/api/v1/workflows/<run-id>/status \
  -H "Authorization: Bearer $YAPPC_TOKEN" \
  | jq '{status, completedAt, error}'
```

## FAQ

**Q: What is the difference between INTENT and SHAPE?**  
INTENT is about *what you want to build* — captured in natural language. SHAPE is about *how you plan to build it* — domain models, API contracts, and system design.

**Q: Can I skip a lifecycle phase?**  
No. Each phase gate enforces that the previous phase completed successfully. This is intentional — it prevents teams from jumping to code generation without a validated domain model.

**Q: How does tenant isolation work?**  
Every API call carries a `TenantContext` (derived from the JWT claims or the `X-Tenant-Id` header). All persistence operations in Data-Cloud scope by `tenantId`, so data from one tenant can never bleed into another.

**Q: How do I add a custom AI model?**  
Implement the `ModelAdapter` interface in `core/ai` and register it with the `AIModelRouter` via the `AIRouterConfig`. See [LLM_INTEGRATION_GUIDE.md](../LLM_INTEGRATION_GUIDE.md).

**Q: What happens if the LLM is unavailable when generating suggestions?**  
`AISuggestionService` propagates the exception as a failed `Promise`. The `SuggestionPanel` frontend displays the error state with a retry option. No silent failures.

**Q: How are approval requests stored durably?**  
`JdbcHumanApprovalService` persists to the `approval_requests` table in the tenant-scoped PostgreSQL schema, and also emits an AEP event so downstream subscribers can react.
