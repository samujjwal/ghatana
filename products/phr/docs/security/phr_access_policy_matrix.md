## 1. Personas

| Persona | Description | Auth Method |
|---------|-------------|-------------|
| `patient` | The individual whose PHR data this is | Username/password + optional biometric (mobile) |
| `provider` | Clinician or healthcare professional with delegated access | Session token + X-Role header |
| `caregiver` | Designated carer with delegated access to dependents | Session token + X-Role header |
| `fchv` | Female Community Health Volunteer (FCHV) | Session token + X-Role header |
| `admin` | System/product administrator | Session token + X-Role header (admin) |

## 2. Route-Level Access Policy

Every request to the PHR backend is enforced by `PhrRouteSupport.requireContext()` which validates that the session headers (`X-Tenant-Id`, `X-Principal-Id`, `X-Role`) are present and non-blank.

| Route Prefix | Allowed Roles | Authorization Rule |
|---|---|---|
| `GET /fhir/**` | patient | Principal must be the resource owner or have a valid Consent grant |
| `GET /patients/**` | patient, provider | Provider must have active consent from patient |
| `GET /clinical/**` | patient, provider | Provider must have active consent |
| `GET /consents/**` | patient, provider, admin | Patient sees own grants; provider sees grants from their patients; admin sees all |
| `POST /consents/grants` | patient | Only patient can issue consent grants |
| `POST /consents/grants/:id/revoke` | patient, admin | Patient revokes own grants; admin may revoke for governance |
| `POST /emergency/access` | patient, provider | Any registered user may initiate emergency access; PHI returned only after reason is recorded |
| `GET /emergency/reviews/pending` | admin | Admin-only |
| `POST /emergency/reviews/:eventId` | admin | Admin-only |
| `GET /provider/patients` | provider | Scoped to the provider's assigned roster |
| `GET /caregiver/dependents` | caregiver | Scoped to the caregiver's enrolled dependents |
| `GET /fchv/dashboard` | fchv | Scoped to the FCHV's community |
| `GET /mobile/dashboard` | patient | Session-header auth; mobile session only |
| `GET /notifications` | patient, provider | Scoped to the caller's principal ID |
| `GET /audit/events` | admin, patient | Patient sees own audit trail; admin sees tenant-scoped events |
| `GET /release-readiness` | admin | Admin-only gate for release cockpit |
| `PUT /profile` | patient | Only the principal may update their own profile |
| `POST /auth/login` | (unauthenticated) | Returns a session token on success |
| `POST /auth/logout` | any authenticated | Invalidates the caller's session |

## 3. Consent Enforcement Model

- Consent grants are recorded in the PHR backend as FHIR `Consent` resources.
- A provider accessing `GET /patients/:id` or `GET /clinical/**` for a patient must have an active consent grant where:
  - `recipientId` matches the provider's `principalId`
  - `scope.resourceTypes` includes the requested resource type
  - `expiresAt` is in the future (or absent for open-ended grants)
- Consent enforcement is checked by `PhrRouteSupport.requireContext()` and the relevant route handler.
- **Cache invalidation**: Consent revocations must invalidate any server-side or CDN cache for the affected patient's resources within 60 seconds.

---

## 4. Emergency Access (Break-Glass)

- Emergency access bypasses standard consent checks.
- The requestor **must** provide a documented reason (backend persists this as an audit event).
- On mobile, biometric authentication is re-challenged before the break-glass PHI is returned.
- All emergency access events appear in the admin emergency review queue (`GET /emergency/reviews/pending`).
- Unreviewed emergency access events trigger an admin notification after 24 hours.
- See [phr_emergency_access.md](./phr_emergency_access.md) for the operational runbook.

## 7. Security Assumptions and Invariants

1. All transport is TLS 1.2+. PHR is never served over plain HTTP in staging or production.
2. Session tokens are short-lived (â‰¤ 1 hour). Refresh tokens are stored in `HttpOnly` cookies on web or in `expo-secure-store` on mobile.
3. Mobile PHI at rest is encrypted with AES-256-GCM. The key lives in the device SecureStore; the ciphertext lives in AsyncStorage. See [phr_mobile_offline_phi.md](./phr_mobile_offline_phi.md).
4. `X-Tenant-Id` is always validated against the session's tenant claim. Cross-tenant access is denied at the `requireContext()` boundary.
5. Pagination parameters (`page`, `pageSize`) are validated; `pageSize` is capped at 100 server-side.
