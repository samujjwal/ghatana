# Digital Marketing Runtime Module Architecture Decision

## Decision
**runtimeModule: false is CORRECT and INTENTIONAL for Digital Marketing**

## Date
2026-05-23

## Context
The Digital Marketing Operating System (DMOS) uses a standard web API product lifecycle profile with a kernel bridge adapter. The `runtimeModule` conformance field in the canonical product registry is set to `false` because DMOS does not require a dynamically loadable runtime module.

## Rationale

### 1. Standard Web API Architecture
Digital Marketing follows the `standard-web-api-product` lifecycle profile, which:
- Uses Gradle build system for Java backend (dm-api)
- Uses pnpm/Vite for React frontend (ui)
- Deploys as container images via docker-buildx adapter
- Does not require dynamic runtime module loading

### 2. Kernel Bridge Adapter Pattern
DMOS implements lifecycle integration through the kernel bridge adapter:
- **Location**: `products/digital-marketing/dm-kernel-bridge`
- **Implementation**: `DigitalMarketingKernelBridge` class
- **Purpose**: Bridges DMOS to kernel lifecycle without requiring a runtime module

The bridge adapter provides:
- Health endpoint exposure (`/health/live`, `/health/ready`)
- Lifecycle phase coordination
- Artifact manifest generation
- Deployment manifest production
- Verification health reporting

### 3. Runtime Module Use Cases
Runtime modules are required for products that:
- Need dynamic loading/unloading at runtime
- Require hot-swapping of components
- Have plugin-based architecture with runtime discovery
- Need to be managed as independent runtime units

DMOS does not have these requirements:
- Components are statically linked at build time
- No hot-swapping needed (standard deployment model)
- No plugin architecture requiring runtime discovery
- Managed as a monolithic deployment unit (api + web)

### 4. Lifecycle Architecture Proof

#### Build Phase
```
Gradle build → JAR artifact → Container image → Artifact manifest
```
- No runtime module needed: standard Gradle build produces JAR
- Bridge adapter generates artifact manifest

#### Package Phase
```
Container buildx → Docker image → Image scan → Package manifest
```
- No runtime module needed: standard container packaging
- Bridge adapter validates container integrity

#### Deploy Phase
```
Kubernetes manifest → Deployment → Health checks → Deployment manifest
```
- No runtime module needed: standard Kubernetes deployment
- Bridge adapter monitors health and produces deployment manifest

#### Verify Phase
```
Health endpoint checks → Verify health report → Lifecycle health snapshot
```
- No runtime module needed: standard HTTP health checks
- Bridge adapter coordinates verification

## Alternatives Considered

### Alternative 1: Implement Runtime Module
**Rejected** because:
- Adds unnecessary complexity
- No business requirement for dynamic loading
- Standard web API pattern is sufficient
- Would require significant refactoring with no benefit

### Alternative 2: Mark runtimeModule: true with placeholder
**Rejected** because:
- Would be misleading (no actual runtime module)
- Would require implementing runtime module infrastructure
- Violates "no mocks/stubs/placeholders" principle

## Decision Impact

### Positive Impacts
- Simpler architecture aligned with standard web API patterns
- Faster build and deployment (no runtime module overhead)
- Easier to understand and maintain
- Consistent with other standard web API products

### No Negative Impacts
- All lifecycle phases work correctly without runtime module
- Kernel integration works through bridge adapter
- No missing functionality or capabilities

## Validation

### Automated Tests
- `DigitalMarketingLifecycleGateTest` validates all required gates
- Bridge adapter tests verify lifecycle integration
- Health endpoint tests verify monitoring

### Manual Validation
- Local deployment works with `compose-local` adapter
- Build/package/deploy/verify phases produce correct manifests
- Health checks respond correctly
- Gate validation passes

## Conclusion

**runtimeModule: false is the CORRECT configuration for Digital Marketing.**

The product achieves full lifecycle integration through the kernel bridge adapter pattern, which is the appropriate architecture for a standard web API product. Implementing a runtime module would add unnecessary complexity without providing any benefit.
