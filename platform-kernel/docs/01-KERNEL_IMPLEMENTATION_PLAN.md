## Current state diagnosis

The repo already knows this problem exists. `docs/README.md` says documentation authority should be centralized, but it still lists audit reports, remediation implementation plans, TODO lists, and closure evidence as “Core Documents,” which directly conflicts with the desired “current vision/architecture/design/ADR/operations only” model. 

The current root `README.md` is also a cleanup target. It presents “Ghatana - Simplified Architecture,” hardcoded historical metrics, a `ghatana-new/` directory tree, and migration status/next steps, which reads like an implementation-session report rather than the current canonical entry point.   

There is already a large cleanup candidate manifest, `md-cleanup-candidates-pass2.txt`, listing exactly the kinds of files that should be removed or consolidated: kernel legacy docs, migration/status docs, generated docs, implementation progress files, infrastructure audit reports, product checklists, session notes, completion reports, verification reports, and many YAPPC archived audit/progress files.    

The repo has useful guardrails, but they are not yet aligned. `config/documentation-truth-scope.json` says historical docs remain reference material and are not release truth sources, but it also excludes `docs/runbooks/` even though operations/runbooks are one of the doc categories we want to keep current.  

The current cleanup gate intentionally excludes `docs`, `code-audits`, `scripts`, `templates`, tests, fixtures, and generated paths, so it cannot enforce the cleanup you want for docs/scripts today. 

Script sprawl is real. `package.json` contains a large mixed command surface: required checks, product checks, evidence generators, architecture checks, release checks, kernel/YAPPC/Data Cloud checks, behavioral proofs, and many product-specific commands. This needs a registry and ownership model, not ad hoc deletion.    

There are clear examples of stale/destructive product-local scripts. `products/yappc/scripts/organize-docs.sh` hardcodes old YAPPC file moves into audit/development/deployment/API folders, and `products/yappc/scripts/cleanup-unnecessary-files.sh` deletes old reports, TODO directories, backup files, and temporary folders with fixed paths. These should not remain as reusable scripts after the cleanup is complete.     

There are also correctness issues inside canonical docs. `docs/GOVERNANCE.md` has a local absolute path reference to `/Users/samujjwal/...`, which the doc truth rules are supposed to prevent. 

Data Cloud has two canonical-looking indexes that disagree on paths. `products/data-cloud/docs/README.md` points to `product/01_data_cloud_unified_vision_market_positioning.md`, while `CANONICAL_DOCS_INDEX.md` points to `docs/product/01_data_cloud_vision.md`. That kind of mismatch is exactly what confuses readers and should be resolved before deleting aggressively.  

## Target state

Keep only these doc classes:

1. **Root entry docs**: `README.md`, `docs/README.md`, `CONTRIBUTING.md`, `LICENSE`, current onboarding/build/test docs.
2. **Canonical monorepo docs**: vision, architecture, governance, build, testing, onboarding, product registry/domain registry explanation.
3. **ADRs**: keep all ADRs, but mark superseded ADRs explicitly instead of deleting architectural history.
4. **Operations/runbooks**: keep only current operational guides, rollback/runbook/security/privacy/incident guides.
5. **Product canonical docs**: per active product, keep a small standard set: `00-VISION.md`, `01-ARCHITECTURE.md`, `02-DESIGN_OR_UX.md`, `03-API_CONTRACTS.md`, `04-TESTING.md`, `05-OPERATIONS.md`, plus ADRs when needed.
6. **Generated docs**: keep only if they are deterministic, generated by a committed command, and either regenerated in CI or clearly marked generated. Otherwise delete.
7. **Module-local README/OWNER**: keep only when current and useful for developers.

Delete or migrate out of repo:

* session notes, completion reports, progress reports, old audit outputs, “final status,” “verification report,” “implementation complete,” “temporary plan,” “roadmap tracker,” stale TODOs, one-off checklists, archived prompt files, backup/migration status docs, duplicated root-level redirect docs, and stale generated docs.
* old docs under `docs/archive/**` and `products/**/docs/archive/**` unless they are ADR-grade historical decisions.
* cleanup candidate manifests after they are converted into an executable cleanup plan.

For scripts, keep only these classes:

1. **CI entrypoints** used by workflows or `package.json`.
2. **Developer commands** documented in root/product README files.
3. **Deterministic checkers** with tests and stable config.
4. **Deterministic generators** whose output is either committed intentionally or ignored intentionally.
5. **Release/ops scripts** with clear owner, inputs, dry-run mode, idempotency, and rollback behavior.

Delete or consolidate:

* one-off cleanup scripts, migration scripts after migration is complete, scripts that move/delete hardcoded old files, duplicate product-local checkers covered by root checkers, unreferenced shell scripts, and scripts whose only output is stale evidence or report files.

## Final cleanup policy

Adopt this rule permanently:

> The repo must not keep working-session documents. Current truth belongs in canonical docs, decisions belong in ADRs, operations belong in runbooks, and implementation work belongs in issues/tasks/code—not stale Markdown reports.

For scripts:

> The repo must not keep one-off scripts after the one-off work is complete. Every script must be registered, owned, referenced, repeatable, and either tested or intentionally documented as a developer helper.

That gives us a clean, current, low-confusion repo: docs become authoritative, scripts become trusted tooling, and temporary work products stop accumulating.
